ETT - A Golang Web Framework

ETT means one in Swedish. This is also the first web framework I learned to implement.

Implementation of ETT can be found here.

ETT is an implementation based on the well-known framework Gin.

ETT contains the following components.

  1. Context: encapsulate Request and Respons.
  2. Trie Tree: achieve dynamic routing.
  3. Route Group Control: achieve group control.
  4. Middleware: support middleware.

Details of ETT

ServeHTTP Interface

ServeHTTP is an interface. When using http package, it requires an address to listen on and a handler to handle requests.

All HTTP requests will be handled if we implements the ServeHTTP interface and pass corresponding instance of handler to ListenAndServe function.

If no handler found, it will use DefaultServeMux, otherwise, our implementation of handler will be called.

Context

Context encapsulates Request and Response as well as supports JSON and HTML type. The reason behind Context is that in some scenarios it will become repetitive and cumbersome to create Header and Body of HTTP responses.

Context is created with each request and destroyed at the end of the request. All information strongly related to the current requests is carried by Context. This design keeps the scalability and complexity internal and simplifies the interface externally.

Context is like Doraemon’s pocket, you can find anything you want in it.

Dynamic Route

In addition to fixed routes, dynamic routes are widely used, for example, /hello/:name can match /hello/world, /hello/Tim, etc.

Trie is an advanced data structure that can be used to achieve dynamic route.

For example, we define the following rules.

  • /:lang/doc, /:lang/tutorial, /:lang/intro

  • /about

  • /tag/data, /tag/string

We can use Trie to represent them as shown in the following figure.

When we need to find an endpoint, start from the root node, and if none of the nodes at an intermediate level satisfy the condition, then it means there is no matched route.

Implementation of Trie

Nodes are designed as shown below.

pattern is the route to be matched, e.g. /index/:lang.

part means the part of pattern, e.g. /:lang.

children stores the child nodes.

isWild denotes if it needs wildcard match.

  • Registration of route

    When developing, routes need to be registered and later will be matched and find corresponding handlers. Therefore, Trie supports insert and search functions.

    • Insert

      Recursively find nodes at each level of the tree, and if there is no match for the current part, create a new node.

    • Search

      Recursively queries the nodes at each level, with exit rules as, match to *, match to nil, or match to the len(parts) level node.

  • Router

    Router is used to add and get routes.

    roots stores the root node of Trie, and use handler to store the HandlerFunc of each request method.

    The return value of getRoute is a map. For example, /p/go/doc matches /p/:lang/doc, so the return value is {lang: "go"}.

Route Group Control

In real developing scenarios, a set of routes usually need similar processing. For example, routes starting with /admin need authentication, etc. In our implementation, we support group control distinguished by prefixes and also nesting of groups.

Middleware

ETT framework needs to have a function that allows users to define their own functionality to be embedded in the framework as if the functionality is natively supported by the framework.

The definition of ETT’s middleware is consistent with the Handler for route mapping, where the input for processing is Context. The insertion point is that after the framework receives the request to initialise the Context, it allows the user to use their own defined middleware to do some additional processing, such as logging, and secondary processing of the Context. In addition by calling the (*Context).Next() function, the middleware can wait for the end of the user-defined Handler processing, and then do some additional operations, such as calculating the time used for this processing and so on. That is, ETT’s middleware supports the user to do some additional operations before and after the request is processed. Meanwhile, it supports setting up multiple middleware and calling them in sequence.
